Zero Trust Network Architecture in Cisco Packet Tracer

A segmented enterprise network with VLANs, ACL-based microsegmentation, DHCP snooping, port security, and a DMZ. Designed and implemented in Cisco Packet Tracer to demonstrate Zero Trust principles.

Project Type: Individual capstone
Technologies: Cisco 2911, 2960 switches, VLANs, OSPF (single site), ACLs, NAT, Syslog, AAA (local), SSH

Project Summary

The goal of this project was to redesign a flat campus LAN into a Zero Trust–inspired network. Each department now lives in its own VLAN with tightly controlled access enforced at the router using inbound ACLs. A dedicated DMZ hosts public web services, and the internal management plane is isolated on its own subnet.

VLAN Segmentation DMZ ACL Microsegmentation AAA + SSH DHCP Snooping Port Security

Objectives

The project follows the “never trust, always verify” mindset. Instead of one big trusted LAN, users, servers, and guests are separated and must be explicitly allowed to talk to each other.

Project Goals

  • Design a realistic multi-VLAN enterprise network.
  • Enforce Zero Trust microsegmentation using inbound ACLs on R1.
  • Isolate public services in a dedicated DMZ VLAN.
  • Secure management access using SSH and local AAA on the router and switches.
  • Detect and block rogue devices with DHCP Snooping and Port Security.

Scope & Constraints

  • Single physical site using Cisco Packet Tracer 8.x.
  • No real VPN clients or RADIUS/TACACS+ (Packet Tracer limitation).
  • AAA implemented with aaa new-model and local user accounts.
  • IPv4-only; IPv6, 802.1X, and full endpoint compliance are out-of-scope.
Note: Even with these limits, the network still demonstrates strong Zero Trust concepts: least privilege, explicit verification, and continuous monitoring.

Network Architecture

Network Topology

The full enterprise Zero Trust topology designed and implemented in Cisco Packet Tracer. Each department, server group, and DMZ zone is isolated using VLANs, ACL microsegmentation, and strict access control.

Diagram Overview: HR, Finance, Users, Guest, Management, Server, and DMZ networks each reside in isolated VLANs. R1 routes traffic using subinterfaces, NAT provides Internet access, and ACLs enforce strict Zero Trust communication paths.

The design uses a Cisco 2911 router for inter-VLAN routing and Internet access, with two Cisco 2960 switches providing user and server connectivity. VLANs separate HR, Finance, Users, Management, Servers, Guest, and the DMZ.

Inter-VLAN Routing: R1 uses GigabitEthernet0/0.x subinterfaces (one per VLAN) with 802.1Q tagging to route traffic between segments while enforcing ACLs on each interface.

DMZ Design

VLAN 70 hosts a public-facing web server reachable from the Internet, but isolated from the internal LAN. ACLs on the DMZ subinterface allow:

  • HTTP/HTTPS from any external client to 192.168.70.10
  • ICMP for basic reachability tests
  • SSH from the Management VLAN (192.168.30.0/24) only

All other inbound and outbound traffic from the DMZ is denied by default, preventing pivoting into internal networks.

Internet Edge (NAT)

A simulated ISP is connected to R1 via GigabitEthernet0/1 using public IP 203.0.113.2/24. R1 performs PAT (NAT overload) for all 192.168.x.x addresses:

access-list NAT_INSIDE permit 192.168.0.0 0.0.255.255
ip nat inside source list NAT_INSIDE interface Gi0/1 overload

This allows internal hosts to browse the Internet while hiding private addresses behind a single public IP.

Security Controls

Multiple layers of security work together: segmentation (VLANs), policy enforcement (ACLs), device-level protections (Port Security, DHCP Snooping), and secure management (AAA + SSH + Syslog).

ACL Microsegmentation

Every user-facing VLAN has an inbound ACL on its R1 subinterface. For example, the USERS ACL (ACL_USERS_IN) allows:

  • DNS + HTTP/HTTPS to internal servers
  • General Internet access via NAT
  • ICMP to specific servers for testing

Traffic to other internal subnets is denied by default, preventing lateral movement between departments.

! Example (simplified)
permit tcp 192.168.10.0 0.0.0.255 host 192.168.20.22 eq www
deny  ip 192.168.10.0 0.0.0.255 192.168.0.0 0.0.255.255
permit ip 192.168.10.0 0.0.0.255 any

DHCP Snooping & Port Security

DHCP Snooping is enabled on both switches for all VLANs. Only the uplink to R1 is trusted; all access ports are untrusted. A rogue DHCP server connected to the Guest network cannot hand out IP addresses.

Port Security restricts each access port to a single MAC address, detecting unauthorized devices and limiting switch-cam attacks.

ip dhcp snooping
ip dhcp snooping vlan 10,20,30,40,50,60,70
interface Fa0/1
  ip dhcp snooping trust

Secure Management Plane (AAA + SSH)

Telnet and HTTP are disabled. R1 and both switches use SSHv2 with local AAA:

  • aaa new-model
  • Login and exec authorization via local user database
  • VTY lines restricted to SSH only
aaa new-model
aaa authentication login default local
username admin privilege 15 secret <password>
line vty 0 15
  transport input ssh
  login authentication default

Only the Management VLAN can SSH into network devices, enforcing role-based access to configuration.

Monitoring with Syslog

A central Syslog server (on the Servers VLAN) receives logs from R1 and both switches, including:

  • Interface up/down events
  • Port Security violations (where PT supports them)
  • Login attempts and config changes
logging 192.168.20.20
service timestamps log datetime msec
logging on

Testing & Validation

The network was validated through structured tests focused on segmentation, access control, rogue device behavior, and secure management.

Segmentation & Access Tests

  • All hosts successfully obtained IPs via DHCP and reached their default gateways.
  • USERS could reach internal DNS / web servers and the Internet.
  • HR and Finance could reach the Internet but not each other.
  • Guest could reach only the Internet (no 192.168.x.x access).
  • Admin (MGMT) could SSH into R1 and switches; other VLANs could not.
Key result: No lateral movement was possible across departments; all cross-VLAN access had to be explicitly permitted.

Rogue Device & DMZ Tests

  • A rogue DHCP server connected to the Guest VLAN failed to assign addresses; legitimate bindings still appeared in the show ip dhcp snooping binding table.
  • The DMZ web server was reachable via HTTP/HTTPS from external clients but could not initiate connections into internal VLANs.
  • NAT translations were confirmed with show ip nat translations, mapping 192.168.x.x hosts to 203.0.113.2.

Limitations

  • Packet Tracer does not support full VPN clients, RADIUS/TACACS+, or endpoint posture checks; remote-access Zero Trust is not modeled.
  • Port Security behavior (err-disable on violation) is inconsistent in the simulator compared to real IOS.
  • AAA is limited to local accounts; centralized identity is future work.
